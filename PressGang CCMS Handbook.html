1. Document Conventions

This manual uses several conventions to highlight certain words and phrases and draw attention to specific pieces of information.
In PDF and paper editions, this manual uses typefaces drawn from the Liberation Fonts set. The Liberation Fonts set is also used in HTML editions if the set is installed on your system. If not, alternative but equivalent typefaces are displayed. Note: Red Hat Enterprise Linux 5 and later includes the Liberation Fonts set by default.

1.1. Typographic Conventions

Four typographic conventions are used to call attention to specific words and phrases. These conventions, and the circumstances they apply to, are as follows.
Mono-spaced Bold
Used to highlight system input, including shell commands, file names and paths. Also used to highlight keys and key combinations. For example:
To see the contents of the file my_next_bestselling_novel in your current working directory, enter the cat my_next_bestselling_novel command at the shell prompt and press Enter to execute the command.
The above includes a file name, a shell command and a key, all presented in mono-spaced bold and all distinguishable thanks to context.
Key combinations can be distinguished from an individual key by the plus sign that connects each part of a key combination. For example:
Press Enter to execute the command.
Press Ctrl+Alt+F2 to switch to a virtual terminal.
The first example highlights a particular key to press. The second example highlights a key combination: a set of three keys pressed simultaneously.
If source code is discussed, class names, methods, functions, variable names and returned values mentioned within a paragraph will be presented as above, in mono-spaced bold. For example:
File-related classes include filesystem for file systems, file for files, and dir for directories. Each class has its own associated set of permissions.
Proportional Bold
This denotes words or phrases encountered on a system, including application names; dialog box text; labeled buttons; check-box and radio button labels; menu titles and sub-menu titles. For example:
Choose System → Preferences → Mouse from the main menu bar to launch Mouse Preferences. In the Buttons tab, click the Left-handed mouse check box and click Close to switch the primary mouse button from the left to the right (making the mouse suitable for use in the left hand).
To insert a special character into a gedit file, choose Applications → Accessories → Character Map from the main menu bar. Next, choose Search → Find… from the Character Map menu bar, type the name of the character in the Search field and click Next. The character you sought will be highlighted in the Character Table. Double-click this highlighted character to place it in the Text to copy field and then click the Copy button. Now switch back to your document and choose Edit → Paste from the gedit menu bar.
The above text includes application names; system-wide menu names and items; application-specific menu names; and buttons and text found within a GUI interface, all presented in proportional bold and all distinguishable by context.
Mono-spaced Bold Italic or Proportional Bold Italic
Whether mono-spaced bold or proportional bold, the addition of italics indicates replaceable or variable text. Italics denotes text you do not input literally or displayed text that changes depending on circumstance. For example:
To connect to a remote machine using ssh, type ssh username@domain.name at a shell prompt. If the remote machine is example.com and your username on that machine is john, type ssh john@example.com.
The mount -o remount file-system command remounts the named file system. For example, to remount the /home file system, the command is mount -o remount /home.
To see the version of a currently installed package, use the rpm -q package command. It will return a result as follows: package-version-release.
Note the words in bold italics above — username, domain.name, file-system, package, version and release. Each word is a placeholder, either for text you enter when issuing a command or for text displayed by the system.
Aside from standard usage for presenting the title of a work, italics denotes the first use of a new and important term. For example:
Publican is a DocBook publishing system.

1.3. Notes and Warnings

Finally, we use three visual styles to draw attention to information that might otherwise be overlooked.

Note

Notes are tips, shortcuts or alternative approaches to the task at hand. Ignoring a note should have no negative consequences, but you might miss out on a trick that makes your life easier.

Important

Important boxes detail things that are easily missed: configuration changes that only apply to the current session, or services that need restarting before an update will apply. Ignoring a box labeled 'Important' will not cause data loss but may cause irritation and frustration.

Warning

Warnings should not be ignored. Ignoring warnings will most likely cause data loss.

1.2. About PressGang CCMS

PressGang CCMS is a Component Content Management System (CCMS) written in Java EE 6 and deployable to JBoss Enterprise Application Platform 6. PressGang CCMS is in active development by the Red Hat Engineering Content Services (ECS) Tools Team.

Note

PressGang CCMS is not a Content Management System (CMS), like Drupal or MediaWiki. A CMS manages published content like wiki articles or blogs, but a CCMS manages the building blocks for content.
PressGang CCMS is only one part of the toolchain used by Red Hat Content Services. It works in conjunction with another tool called Publican. Publican is the tool used to build human-readable outputs such as HTML and PDF from Docbook.
History
In the early days of content authoring at Red Hat, the team started out by using SGML and Makefiles. The next step was to develop Publican, which greatly simplified the publication process and allowed for localization.
The first attempt at topic reuse was a tool called Topic Tool. It was a prototype system that used Docbook's <xi:include> tag to build up modular documentation. In reality it was difficult to use and maintain.
PressGang CCMS simplifies the toolchain, and makes topic reuse and modular documentation more feasible.

In April 2013 we undertook an evaluation of Calenco XML CMS. It was found to underperform PressGang CCMS significantly, and lacked the flexibility needed for Red Hat's workflow. It also has a mixed license model which is not appealing. Certain parts of Calenco are open source, but others are proprietary, and are licensed from a company called XML Mind. If we wanted to go this route, we could license the components directly as well.

Rather than spend thousands of dollars per year on a proprietary system, with huge content-conversion and re-tooling costs, we have built PressGang CCMS. The goal is to release it into the community using the same model as Publican, so that others can receive the same benefit that Red Hat is already receiving from it.
PressGang CCMS may not be the final solution. It is impossible to know what Red Hat will be using to manage documentation in 5 or 10 years. But PressGang CCMS is the next step, to enable Content Services to scale with Red Hat's growth and to produce great documentation efficiently.
Purpose
PressGang CCMS was created to solve some specific problems within Content Services, and also to provide some major benefits.
  • The first goal of PressGang CCMS was to enable topic reuse. As Red Hat grows its product line, and the work load of ECS grows, it is clear that we need a way to reuse relevant content between multiple groups. Even if content is reused only once, it is still an increase in efficiency. With the approach that Red Hat takes toward product layering and interoperability, it is clear that topic reuse is something that will benefit all of us.
  • This is even more obvious in the area of localization. PressGang CCMS provides an easy way to push and pull strings from the Zanata Translation Management System, and also allows Zanata's translation memory to be used more effectively on new content. This results in content which is already partially translated before formal translation is even begun.
  • Another goal was toolchain simplification. Red Hat's products are complex enough for new writers to understand, without having to also learn and understand Fedora, RHEL, Docbook, Publican, Subversion, Git, CVS, Brew, RHPkg right away as well. PressGang CCMS gives writers a faster route to productivity. The paradigm of a web-based UI is a very familiar one, and significantly eases the learning curve. This gives writers more bandwidth to learn about the products they will be documenting, and means that they spend more of their time doing the writing that they actually want to do. Writers will still have lots of opportunities to increase their technical skills, but they don't need to do it just to get started.
  • Planning, scoping, and gap analysis are easier to manage with good reporting tools at our disposal. With PressGang CCMS, it is easy to get information about which types of topics are present and absent in documentation, how much writing actually goes into a given release, and how realistic an initial documentation plan actually was, after the fact. The amount of trackable data is huge. Some of the questions that can easily be answered are:
    • How many topics were created/updated/reviewed this week?
    • Have we met our commitments for this release?
    • Do we have any topics about securing the REST interfaces in EAP?
    • How much can a team of three people be expected to write in a 6-month period?
    • Which book has the most bugs logged against it?
  • We need to be able to remix existing content for new formats or audiences. What if we start out with a small product with a few security concerns in a User Guide, but eventually the product grows to the point that we need a dedicated Security Guide? With books in Subversion, this may become an exercise in regular expressions, copy, and paste. With the PressGang CCMS, this is a matter of searching for security topics related to Product X, and asking the CCMS to create a new content specification including the results of the search. The content specification can now be organized as appropriate.
    The above example actually happened, very recently. The Security Guide took a couple of hours to reorganize and publish to an internal staging site. It was able to be reviewed before lunch time of the same day it was started.
  • We need to work with several different upstreams. We can't guarantee that an external team will use Subversion, Git, or another specific system. We can't even guarantee that they will use Docbook. We need a system that lets us add functionality when we need it, so that we can react quickly to changing business environments.
Architecture
PressGang CCMS consists of several different components which work together. These are the web-based user interface (the UI), the REST API, a database to hold information about topics, a command-line tool called the CSProcessor, an integrated instance of the BIRT open-source reporting tool, and the DocBuilder continuous-intergration server. These components will be explained in detail later in this guide, but here is a quick overview of each.
UI
The UI is the most common entry-point to using the CCMS. It includes facilities for creating, updating, viewing, and searching for topics and content specifications. It is accessed via a web browser. The UI is the way that most writers will interact with the CCMS.
REST API
A REST API is a way of interacting with a web service by passing specific URLs and parsing the results. This is a very common way to interact with web-based applications. The PressGang CCMS uses a REST API to pull together all of its different components, such as the database and the UI.
The REST API also allows other applications, services, and scripts to be built to extend the functionality of the CCMS. One such extension is the PressStar editor (https://github.com/jwulf/press-star, which was built using the Node.JS JavaScript library. The REST API gives the CCMS the ability to grow and change with the needs of Red Hat.
WebDAV
Topics are exposed via a WebDAV filesystem, which can be mounted as a drive in any modern operating system, and allows topics to be edited with regular text editors.
Database
PressGang uses a MySQL database to store all topics, content specifications, metadata, and revision history information. The important thing about this is that it is a relational database (RDBMS). Although it is using MySQL at the moment, it is not tied to any specific RDBMS.
CSProcessor
The basic organizational structure of a book built using PressGang CCMS is a content specification map. Currently this map is a plain-text file with special syntax. Content specifications are created and updated using a command-line tool called the CSProcessor. This tool is also used to create the payload which will be built into a book using Publican. In the future, the CSProcessor will be replaced or augmented by a web-based tool. The CSProcessor is used by documentation designers to create and organize the structure of books.
DocBuilder
The DocBuilder is a continuous-integration service which rebuilds content specifications at a regular interval. It is used by different teams for different reasons. Writers use it to preview their work and to collaborate with external teams. Translators use it to view content as they are translating it. The DocBuilder only builds books that have been registered with it. It is much faster and more flexible than the official documentation stage, but it does not output books in exactly the way that customers will see them.
One advantage to the DocBuilder is that its output can be customized to be useful to a particular team, such as translators, QE, or Product Management.
BIRT Reporting Framework
BIRT is an open source reporting framework. The PressGang CCMS includes an integrated BIRT instance, which provides reports about topics, books, releases, and teams. BIRT is flexible enough to be able to provide a wide range of different types of reports.
PressGang CCMS Virtual Application
The Virtual Application is a VM built with VirtualBox that includes all the CCMS components in a single, isolated, ready-to-run environment. It was created to enable external parties to experiment and build upon the base platform. An example of this are university students using the CCMS and its content as the basis for language processing projects.

The PressGang CCMS team is aware of the following issues. Workarounds are provided where relevant.
Author Group Information for Public Consumption
If writers are not correctly assigned to topics, the CCMS inserts a default user of "SkyNet Alpha Build System". This placeholder is not suitable for public documents.

Note

The following processes are work around methods until https://bugzilla.redhat.com/show_bug.cgi?id=981118 is implemented.
To add a writer by name, follow these steps:
  1. Assign a registered CCMS user (writer) to a topic.
  2. On the Extended Properties tab of each topic, add the First Name and Last Name fields.
  3. In each topic, add the actual writer's first and last name to the fields.
To add a Red Hat Documentation Group author:
  1. Assign a registered CCMS user to a topic.
  2. On the Extended Properties tab of the topic, add the First Name and Last Name fields.
  3. Add the string Red&nbsp;Hat Documentation Group to the Last Name field.

    Note

    Adding this string to the last name field ensures the string is not reordered when translated to languages such as Japanese.
Limited Support for Docbook 5 Topics
Topics were only recently introduced as a DocBook element in version 5.1, and Docbook 5 is not officially supported by Publican 3 (see BZ#697367). Section elements have been used to emulate a topic, but this has led to a number of limitations, many of which are listed below.
Entity files need to be stored separately.

Note

This problem is a very low priority, because the usage of entities, beyond the ones required by some Red Hat Publican brands, is discouraged within ECS. Entities create problems with translation and content reuse.
The CCMS does not yet have a mechanism for storing or updating entity files (typically book_name.ent). To work around this issue, you can store the entity files in an external filesystem (and back them up in a VCS), and direct the csprocessor command to include them at build time. This problem will be fixed at a later date.
Topics cannot include nested <section> elements
Because the CCMS adds some content to the end of each topic (such as the Report a Bug links), allowing nested <section> elements causes validation errors.
This is easy to work around. Make the top-level section for each level a Section: in the content specification, rather than a separate topic. In this way, you can nest as deeply as necessary.
Real-time "help desk" support is not available.
The entire ECS Tools team works in Brisbane, and are only available to answer questions during that time. To work around this limitation, use Bugzilla or one of the other feedback mechanisms mentioned above to ask questions and provide feedback.

2.1. About Topic-Based Authoring

Topic-based authoring is a methodology for distilling content into topics, which can stand somewhat on their own, and then building a document out of these topics. Topic-based authoring is different from narrative authoring, which assumes that a book or article will be read cover-to-cover.
Darwin Information Typing Architecture (DITA) is the most well-known paradigm for topic-based authoring. Red Hat has borrowed some of the principles of DITA while retaining the Docbook DTD, which provides for a more mature toolchain and richer markup language.
Relationship between PressGang CCMS and Topic-Based Authoring
PressGang CCMS does not force a topic-based approach, and a topic-based approach can be used outside the scope of PressGang CCMS. However, the CCMS was designed with a topic-based approach in mind, and works well with this approach.

Table 2.1. Topic-Based vs Narrative Writing

Topic-Based Narrative
Assumes readers may enter the document at any point
Assumes readers will start at the beginning and work their way through the book or article
Allows for repetition of key content at the places in the document where they are needed, without increasing document maintenance
Uses copy-and-paste or cross-linking for content that is relevant in different areas of the same content
Allows existing content to be "remixed" into different formats depending on the audience or other factors
Restructuring of a document requires a lot of work
Provides for different topic types which have different structures and rules (concept, task, and reference)
Treats all content the same, because it's basically all words, right?
Makes it easy for multiple people (and teams) to share and work on common content together, because each "work unit" is small and relatively self-contained
Typically, one person will own one chapter or one book, because this makes for minimal editing conflicts. This makes it harder to share and review work.
Because units of documentation are smaller, bugs can be filed right where the problems are.
Bugs are filed on a per-book basis, with only the bug-filer's description to guide the writer about where to find and fix the problem.

These are only a few of the benefits of topic-based authoring. For more information, refer to http://en.wikipedia.org/wiki/Topic-based_authoring.

Sometimes a task needs to have some information in it that looks conceptual, and sometimes a concept needs to have more information than a simple dictionary definition. A task may not always look like a procedure, but sometimes it may look like an illustrated example or a workflow. When categorizing topics, use your judgment. When segmenting content into topics, think about the unit of granularity that makes the most sense for reuse, not the smallest granularity possible.

2.2. Topic Types

2.2.1. Concept

A concept, in terms of topic-based writing, is explanatory material about something. It is a definition, but contains additional information. A concept may talk about a single thing, or a relationship between things or overview of a thing. A concept often answers the question "What is _____?", but continues on to answer "Why should I care?"
Separating concepts from their associated tasks and references provides a natural separation for a reader between the "What", "Why", "How", and "What Else" questions which are addressed in technical documentation. This separation provides the reader with the confidence that comes with understanding what's involved before jumping into a task.
At Red Hat, topics usually have titles that start with the word About . Here are some examples.
  • About Encryption
  • About Component Content Management Systems
  • About Bedsheets
  • About the Components of a Service-Oriented Architecture
Concepts can contain paragraphs, images, and other types of explanatory content.
Granularity of Concepts
A common trap that people fall into with regard to concepts is making them too granular. Make sure that your concept represents a complete and useful idea to the user. When Red Hat first embarked on the topic-based authoring journey, we attempted to be extremely pure-minded about the content of a concept, and the result was often choppy prose. Over time, these choppy contents are often being merged together into units that make more sense as standalone, reusable units. Conditional statements assist in making concepts more universal without sacrificing cohesion of content. The writer's judgment on behalf of the reader is always the final decider.

Note

This topic itself is a concept.

2.2.2. Task

A task is a topic that answers the question "How?" for the reader. In Docbook, a task is usually either a <procedure>, although sometimes it may be a series of code snippets with explanatory material or an image with explanatory information, such as a workflow or diagram. In published output, a <procedure> is rendered as a numbered list, with optional subtasks or alternative steps.
The title of a task starts with a verb. "Wash a Car" is preferrable to "How to Wash a Car" or "Washing a Car" as a title. Below are some examples of tasks.
  • Make a Bed
  • Install JBoss Enterprise Application Platform 6
  • Brew Good Coffee
  • Configure Apache HTTPD Server
A task often includes an introduction which contextualizes it in the mind of the reader, a list of prerequisites (which may be links to other tasks), and a result statement, which describes the result of performing the task.

Note

Often, a task is the thing that you start out to document. Along the way, you realize that you need some supporting concepts and references to go along with it. A great way to start a documentation plan is to list all of the tasks that need to be documented.

A reference includes extra information about a concept or task. A reference may consist of one or more tables or lists, detailed diagrams with call-outs, or well-commented standalone code examples. 2.3. Metadata

2.3.1. Per-Topic Metadata

Depending on how you look at it, some types of metadata can be stored and tracked without using a CCMS at all. This includes the revision history which is stored in a version-control system and the information which can be gleaned from the specific Docbook markup that is used to tag a document.
If you look specifically at Docbook tagging as metadata, Red Hat has been storing metadata about content for a long time. However, the CCMS lets you go beyond that, and also lets you do more with the metadata itself. Here are a few types of metadata which are tracked by PressGang CCMS and how they can be used. More types of metadata may be added later, and some types which are not considered useful may be removed.
  • Revision info, such as the time, date, writer, and the diff itself
  • Products and versions to which a topic has relevance
  • Content specifications which use the topic
  • Bugs filed against the topic
  • Documentation lifecycle information about a topic, such as expiry or deprecation
  • Status of peer review or QE review on a topic
  • Relevant audiences for a topic
  • Keywords and other types of metadata which may be used by search engines or other external consumers
Some of the ways this metadata can be used include detailed reporting, gap analysis, remixing of content, and documentation lifecycle automation.
Currently, most of this metadata is managed by means of tags, which are checked or unchecked in the UI. The metadata we want and need to capture is under review currently, and may change. The goal is to make the metadata as easy to use as possible, with the least amount of trouble to keep it up to date.
New Metadata System
A new metadata system, which was agreed upon by consensus of CCMS users, is not yet implemented. The purpose of the new system is to minimize the amount of metadata that needs to be managed by hand, and to increase the automation of the process. For now, much of the work that will be automated needs to be done by hand. Process can be tracked at https://bugzilla.redhat.com/show_bug.cgi?id=958331.

3.1. About Topic-Based Authoring

Topic-based authoring is a methodology for distilling content into topics, which can stand somewhat on their own, and then building a document out of these topics. Topic-based authoring is different from narrative authoring, which assumes that a book or article will be read cover-to-cover.
Darwin Information Typing Architecture (DITA) is the most well-known paradigm for topic-based authoring. Red Hat has borrowed some of the principles of DITA while retaining the Docbook DTD, which provides for a more mature toolchain and richer markup language.
Relationship between PressGang CCMS and Topic-Based Authoring
PressGang CCMS does not force a topic-based approach, and a topic-based approach can be used outside the scope of PressGang CCMS. However, the CCMS was designed with a topic-based approach in mind, and works well with this approach.

3.2. About Content Specifications

A content specification refers to the output of the second stage of documentation according to the planning model espoused by Joanne Hackos. The first stage results in an information plan, which is a list of resources and information available. The content specification is a detailed description of a documentation deliverable. Although it may take different forms, in terms of PressGang CCMS, a content specification is a table of contents of a book or article. Its format is both human-and-machine-readable, and the CCMS can process it to add or update topics and metadata. The content specification is also used to build the payload which will be built by Publican and published.
Currently, the content specification is a text file which is parsed by a command-line client called the csprocessor and stored as a topic in the CCMS. In the near future, the content specification will be stored as a database object, and will be able to be manipulated by a GUI tool which will be part of the web-based UI.
Keeping content specifications separate from the topics they incorporate provides a level of abstraction which allows content to be reused (even within the same book) and remixed with minimal maintenance. For instance, a set of security-related topics in different guides can easily be remixed into a stand-alone Security Guide. When a topic is revised, each guide which uses it can be refreshed to use the new content just by a rebuild, with no extra editing required.

3.4.2. Use the csprocessor Command

The csprocessor client is a command-line application, so it must be run in a terminal. For a full listing of the commands accepted by the csprocessor command, use the following command: 
csprocessor --help
This topic gives a brief overview of the most common tasks that are performed using the csprocessor client.

Note

Most of the commands below take either a map ID or a filename as a parameter. For the ones that take a map ID, if you have downloaded a working copy of your book and you are in its directory, you can omit the map ID. Refer to the section below entitled Download a Map and Its Book Into a Project Directory for information about downloading a working copy of a book along with its associated map.
Create an Empty Template of a Map
To begin structuring a new document, first create a template of a map. This template is only stored locally. It contains place-holders for required information, and gives basic syntax guidelines. Use the following command to create an empty template called my_map.contentspec in the current working directory. The name and extension of the file are unimportant. 
csprocessor template -o my_map.contentspec
Validate a Map's Syntax
When you are working on a map locally, you should check the syntax often so that you can catch mistakes early. The csprocessor expects a simple but rigid syntax to be followed. To check the syntax of your map, use the following command: 
csprocessor validate my_map.contentspec
The output will report whether or not the map is valid, and if not, the line where the first problem was found. Edit the file to fix the error and try again.
Create a New Map
After creating your map locally and adding some content to it, use the following command to store the map in the CCMS. Afterward, your local copy of the file can be removed. The syntax of the map is validated before it is allowed to be uploaded. If the map is created successfully, the topic ID of the map is returned. 
csprocessor create my_file.contentspec
Retrieve an Existing Map
After saving your map to the server, you should retrieve it from the server each time you need to work on it, to ensure that your copy is up to date. Use the following command to retrieve the map with the ID 13968 to a local file called 13968.contentspec in the current working directory. 
csprocessor pull 13968 -o 13968.contentspec
Push Changes to a Map
After working locally on an existing map, use the following command to push the changes to the CCMS server. A new revision of the existing map will be created, so that you can revert to a previous revision if needed. 
csprocessor push 13868.contentspec
You do not need to specify the map's ID, because it is included in the map file itself. It is not derived from the file name. The file above could be called foo.contentspec.
This only pushes changes to a map. If you have made changes to a book itself, you can push these changes too, using the csprocessor. Refer to the section below entitled Push Locally-Made Changes to a Book's Files to the CCMS.
Get Information About a Map
To find some statistics about a map, such as how many topics it includes and how many of those topics have at least some existing XML content (have been started), use the following command, which takes the map ID as its parameter. If you are in the book's project directory, you can omit the ID. 
csprocessor info map_id
Download a Map Into a Project Directory
To download a map into a project directory, use the following command. The project directory is created as a subdirectory in the root directory you specified when you configured the csprocessor application, and defaults to the current working directory. It contains the map and a csprocessor.cfg file, which can be used to override the default behaviors of the csprocessor command on a per-project basis. 
csprocessor checkout map_id
If the project has been checked out before, an error is shown. Data is not overwritten.
Customize the Behavior of the csprocessor Command for a Specific Project
After checking out a project with the csprocessor, you can customize which CCMS server is used to manage it, build parameters for Publican, and other options. To do this, edit the csprocessor.cfg file in the project's directory. All of the options in Section 3.4.1, “Install CSProcessor” apply here, along with several book-specific options.
Download the XML of a Book from a Map
To download all of the XML for a book from a given map, use the following command: 
csprocessor build map_id
This downloads the book as a ZIP archive in the current working directory. You can unzip the archive to build and publish it with Publican.
Download and Build a Preview of the Book
To download and build a preview of the book with Publican, use the following command. This command follows the instructions given in the csprocessor.cfg file, which builds a HTML-single output by default. 
csprocessor assemble map_id
To attempt to open the built book in your default web browser, use the preview command instead. This will only work if you are using a windowed environment like GNOME or KDE. 
csprocessor preview map_id
Export a Book with One File Per Chapter, Rather than One File Per Topic
To export a book with one file per chapter, use the --flatten parameter with either of the build or assemble commands.
Override Part of a Book with Local Files
Sometimes you may need to override an automatically-generated file such as the Revision_History.xml, Feedback.xml, Legal_Notice.xml, or Author_Group.xml with a copy that is local or in a VCS which is separate from PressGang CCMS. To do this, use the --override parameter and the name of the automatic file and the path of the file to use instead, when you build the book. The following example overrides the default Revision_History.xml with one stored in /home/misty/. 
csprocessor build --override Revision_History.xml=/home/mistyk/Revision_History.xml map_id
If you are inside a project directory, you can omit the map ID.
You can also use the assemble instruction, rather than build, to build a preview book in the same step: 
csprocessor assemble --override Revision_History.xml=/home/mistyk/Revision_History.xml map_id
Build or Assemble a Translated Book From a Map
By default, the English book is built or assembled. Use the --lang and --target-lang parameters to request a different language instead. The first parameter expects a language code used by Zanata and the second expects a language code used by Publican. They sometimes differ, as in the example below for Japanese. 
csprocessor build --lang ja --target-lang ja-JP map_id
If the book has not yet been translated, or has been partially translated, English strings will be used instead of the target language strings.
Download a Snapshot Which Includes Per-Topic Revisions
You can download a snapshot of the map which includes revision information for each topic. This snapshot will not change even if a topic is subsequently edited. The following command downloads such a "frozen" map." Note that this command does not save the frozen map back to the server. 
csprocessor pull-snapshot map_id
Freeze a Map At A Point In Time
You can use a single command to add per-topic revision information to a map and push this modified map back to the server. This effectively creates a tag of a book, which can be rebuilt and remain the same, even after topics have been subsequently edited. This is important to do before sending a document for translation or publishing the document to the public.
The first variant of this command OVERWRITES the original map with a versioned map. 
csprocessor snapshot map_id
The second variant of the command leaves the original map intact, but creates a new map with the per-topic revisions. This is often the desired behavior, so that work can continue on the original map rather than shifting work to a new map. 
csprocessor snapshot --new map_id
Push a Book's English Strings to Zanata for Translation
To push a book's English strings to the Zanata instance specified in the ~/.config/csprocessor.ini file, which prepares them to be translated, use the following command. In most cases, you should only run this from a "frozen" map (see the snapshot and pull-snapshot commands above). 
csprocessor push-translation map-id
If you are pushing the translation to a Zanata server, then PressGang CCMS requests Zanata to run translation memory over the strings as they are pushed, so that they are partially translated if possible before the translator even sees them.
Search for a Content Specification
To search for a content map with a given string in the Title, Product, Version, or Created By fields of the map, use the following command. This command searches for maps which include the string "CCMS" (case sensitive). This search uses the same API as the search interface in the web-based UI, so it returns the same types of results. 
csprocessor search CCMS
Check the Status of a Local Project Directory Against the Server
To check to see if your local copy of the content map in your project directory is in sync with the server, use the following command within your project directory: 
csprocessor status
The command does not work with individual content specifications, and does not actually look at the map. It only compares the checksums of the local and remote copy.

3.6. CSProcessor Map Workflow

3.6.1. About the Workflow Phases

The eventual intent is to automate a sensible workflow for a CSProcessor map, which aligns with the standard Red Hat content development process. Currently, this is manual, for the most part. The phases are (loosely):

Content Development Phases

Planning
The PRD, ERD, and existing documentation bugs are examined, and the things which will be documented and fixed in the upcoming release are decided, in collaboration with Product Management, Engineering, QE, etc. A Documentation Plan is the first output of this process, and after the Documentation Plan is signed off, basic Content Specifications are the second delivery. Bugs are raised to track each requirement.
Development
In the Development phase, content specifications are fleshed out and content is researched and written. The outputs of the Development phase include draft-quality content, followed by content which can be submitted for testing and review.
Testing
In the Testing phase, the documentation team collaborates with QE and subject-matter experts (SMEs) to find and fix errors and gaps in the documentation. The output of the Testing phase is a series of bug fixes which results in English documentation which is ready to be released.
Localization
Many products at Red Hat have a requirement for documentation to be translated into multiple languages. As soon as the final English documents are complete, a string freeze goes into effect, and the documentation is given to the Localization team for translation. In terms of the CCMS, the content is pushed to Zanata using the CSProcessor command.
Release
In the Release phase, all content is published to the Customer Portal. The Release phase often takes several days, from the green-light to publish, to the public announcement. The actual publication of the documents is done by Release Engineering. The output of the Release phase is published documentation.
Maintenance
The Maintenance phase is the time between releases, and the output of the Maintenance phase is continued bug reporting and feedback, which can be integrated into future releases.
The above process happens many times over the lifecycle of a product.

A topic revision is created every time a topic or its associated metadata is edited and saved to the database. Changes to following properties, fields or relationships will result in a new topic revision being created:

  • Title, description, locale
  • Tags
  • Extended properties
  • Source URLs

3.7.2. Revision-less (Trunk) and Frozen CSProcessor Maps

By default, content specifications use the newest revision for each topic. This is useful when you are in the development phase of your project, as you never need to worry about your new revision showing up in builds.
Before you send your book for translation or publish your content, you need to populate your content specification with revision numbers. This is known as freezing the map. A frozen map will always use exactly the revisions specified in the map, even if newer revisions exist. This is useful for the following reasons:
  • You can always rebuild the exact version of the book that was published.
  • If you need to update a topic post-release, such as a security issue, you can manually update the revision of the specific topic you need to update, without affecting the rest of the book's content.
  • If behavior of a product or technology changes between releases, you can update the topic to include the new behavior, but the old book still uses the old revision, which is relevant to its version of the product.
  • For minor and maintenance releases, localization happen with minimal disruption, while development work begins on the next release.
You can freeze an entire content specification, you can clone and freeze a copy of the content specification and continue working on the existing one as your "trunk", or you can freeze a single topic within an otherwise-unfrozen content specification. Refer to Section 3.4.2, “Use the csprocessor Command” or Section 3.5.3, “Freeze a Map” for more information.

3.7.4. Compare Revisions of a Topic

The diff between two revisions of a topic can be viewed in the user interface. When viewing a topic, select the Revisions button. The Revisions view lists all the revisions of a topic, and clicking the Diff button will display a window with the differences between the revision that is currently being viewed, and the revision whose diff button was clicked.
A topic's XML at a particular revision can also be downloaded through the REST interface. This makes it possible to use any desktop diff tool to view the differences between two revisions of a topic (or the revisions of two different topics).
The script below will display the diff of two topic revisions using the Meld application. 
if [ "$1" == "" ] || [ "$2" == "" ] || [ "$3" == "" ] || [ "$4" == "" ]; then 
  echo This command requires 4 arguments:
  echo     Argument 1: The 1st topic number
  echo     Argument 2: The 1st topic revision
  echo     Argument 3: The 2nd topic number
  echo     Argument 5: The 2nd topic revision
  echo
  echo For example: pressgang-diff 33 377091 33 374525
  exit
fi
T1=$1
R1=$2
T2=$3
R2=$4
Comp1=http://skynet.usersys.redhat.com:8080/TopicIndex/seam/resource/rest/1/topic/get/xml/$T1/r/$R1/xml
Comp2=http://skynet.usersys.redhat.com:8080/TopicIndex/seam/resource/rest/1/topic/get/xml/$T2/r/$R2/xml

# The tool used to display the diffs
DIFFAPP=meld
# The temporary file for the first topic
OUTPUTFILE1=/tmp/pressgangdiff1.xml
# The temporary file for the second topic
OUTPUTFILE2=/tmp/pressgangdiff2.xml

# Download the two topics
wget --output-document=${OUTPUTFILE1} ${Comp1} 
wget --output-document=${OUTPUTFILE2} ${Comp2}

# View their diff
${DIFFAPP} ${OUTPUTFILE1} ${OUTPUTFILE2}
Assuming the script above has been saved into an executable file called pressgang-diff, it can be run with the following command: 
./pressgang-diff 33 377091 33 374525
  • The first parameter, which is 33 in this example, is the topic ID of the first topic to be compared.
  • The second parameter, which is 377091 in this example, is the topic revision of the first topic to be compared.
  • The third parameter, which is 33 in this example, is the topic ID of the second topic to be compared. This will usually be the same as the first parameter, but does not have to be.
  • The fourth parameter, which is 374525 in this example, is the topic revision of the second topic to be compared.

4.2. Considerations for Topic Reuse

While it is understood that not every topic is appropriate for topic reuse, it is important to keep in mind the possibility that another project may benefit from reusing your content in the future. We all have more potential work on our plates than we can accomplish, and reuse helps alleviate this pain. Here are some of the key benefits of topic reuse.
  • Efficiency - reused content is more efficient than rewritten content.
  • Maintenance - reused content only needs to be updated in one place, and the changes can (optionally) propagate through all books which use that content.
Topic Granularity
The most important consideration for topic reuse is topic granularity. Along with the principles of Concept, Task, and Reference, thought needs to go into whether the topic is the right "size." A topic needs to make basic sense on its own, and also be a useful unit of reuse. Of course, this is the ideal situation, and some types of content do not fit directly into this mold.
Useful Tools for Reuse
PressGang CCMS provides some mechanisms which make reuse feasible and low-risk. These features are described briefly below, but are covered more thoroughly elsewhere in this guide.
Topic Revisions
A revision is saved each time a topic is updated. Not only can you revert to a previous topic, but you can also direct your csprocessor map to only use a specific revision of a topic, ignoring all newer revisions. This prevents changes from inadvertently effecting content for which they are inappropriate.
Conditional Build Statements
Docbook includes support for conditional build statements, but this support is rather limited. PressGang CCMS allows for much greater flexibility, by pre-parsing the conditional statements before creating the Publican payload, and by allowing for regular expression (regex) matching of conditional build statements.
Conditional build statements allow content to be reused even if it needs some changes in order to be relevant to a particular product or guide. It also allows you to make changes to a topic without fear of it causing side-effects in other content.

4.3. Typical Writing Workflow

The typical workflow of a writer in the CCMS (as opposed to a document designer or illustrator, even though they may be the same person), is as follows. Variations will arise due to the technical knowledge the writer has of the subject, availability of subject-matter experts (SMEs), and the overall workflow of the individual project.
Research
When you first receive an assignment, you will probably need to research the subject matter. Research includes, but is not limited to:
  • Internet searches
  • Mailing lists and subject-specific web forums
  • Previous documentation
  • Competitors' documentation
  • Reading the source code
  • Trying things out yourself
  • Interviewing subject-matter experts on the phone,video chat, IRC, or email
During the research phase, you should keep track of all source material URLs using the Source URLs tab in the individual topic's view in the CCMS.
Writing
During this phase, you will write the first draft of the content. Inevitably, you will find gaps in your knowledge and research, so there will likely be frequent trips back to the Research phase.
You can keep track of your progress using the DocBuilder continuous integration system.
Review
The Review phase has several parts. At any time during the Review phase, you may need to go back to the Research or Writing phases.
  • Peer Review - you can ask your peers to evaluate your content for style and accuracy.
  • SME Review - you can ask your SMEs to take a look at the output of your content on DocBuilder and let you know if it is accurate and if it has gaps.
  • QE Review - during the formal Testing phase, your product's QE team will review your content for technical accuracy, and file bugs where they are needed. You will have time to address these issues before publication. This may also involve going back to the Research and Writing phases.

4.4.3. Write or Modify the Topic

Writing topics is, obviously, central to the work of Content Services. PressGang CCMS provides several different ways to write your content.
Use the built-in editor
The built-in CCMS editor is part of the web UI. It is an embedded instance of an open source editor called ACE Editor. The full documentation for ACE Editor can be found at http://ace.ajax.org/.

4.4.5. Debug a Build Error in DocBuilder

Most of the time, if you have a syntax error in a single topic, DocBuilder can work around it and build the rest of the book. It leaves an admonition in the failing topic, and a list of errors in an appendix at the end of the book.
Occasionally, you may make a syntax error in a topic, which causes your book to be completely unbuildable. If this happens, when you browse to the book in DocBuilder, instead of the book itself, you will see a directory listing which contains two files: build.log and publican.log. You can use these files to debug the building error.
build.log
The build.log file contains a log of the build and assemble commands, which run on DocBuilder to build your book. Review this output to see if your csprocessor map is valid or there is another at the level of the content specification. At the bottom of the file, you will see output such as the following, which will tell you if there were actually errors building the book with Publican. 
Content Specification successfully built with 3 Errors and 42 Warnings
Output saved to: /tmp/buildbooks13968/Book.zip
publican.log
The publican.log file shows the output of the publican build command. This will show you if the Docbook itself is invalid. Look for the topic ID and the actual error. Note that this error message is formatted in Docbook, because normally it would display in the book's appendix.
Search for and edit the failing topic. Fix the error and save the topic.

4.6.1.2. About Injection

Traditionally, Docbook uses the <xref> tag to cross-reference between content in a single guide. This attribute relies on a stable ID. In a topic-based environment, this mechanism becomes brittle and unreliable.
In PressGang CCMS, each topic has a unique ID. The ID does not change, even if the title or ID of the content does change. For this reason, PressGang CCMS includes a functionality called injections, which allow you to include a link to a topic by ID. The only requirement is that the topic must also be in your content specification. You cannot link to a topic which is not included in your book.
Several types of injections are available for use in a variety of situations.

4.6.1.5. Use Refer-To Links

Refer-to links are specified in the content specification, on a per-topic basis. These links appear at the bottom of a topic, and allow the reader to select where to go next. You can refer the reader to a specific topic, or to a navigation element such as a part, chapter, section, or appendix which is specified in the content specification.

 MyCurrentTopic [15941] [Refer-to: MySecondTopic [15939], MyThirdTopic [58444]]

MyTopic [34433] [Refer-to: T-Chapter-Introduction]

4.6.2. Conditional Build Statements

4.6.2.1. About Conditional Build Statements in the CCMS

Docbook Conditional Statements
Docbook includes conditional building statements already, via a mechanism called profiling. Following is an example of a conditional build statement in Docbook. 
<book>
     <xi:include href="chapt-about.xml" /> <!-- a chapter -->
     <chapter condition="developer">
          <title>Developer Introduction </title>
          <xi:include href="chapt-content-developer_intro.xml" />
 
     </chapter>
     <chapter condition ="administrator">
          <title>Administrator Introduction</title>
          <xi:include href="chapt-content-admin_intro.xml" />
 
     </chapter>
</book>
In this example, the book includes one chapter if the condition developer is met, and another if the condition administrator is met. condition is not the only element available in Docbook profiling.
To build a Publican book with conditional statements, you need to specify the condition for your build using the condition: parameter in the publican.cfg file.
Problems with Docbook Conditional Statements
Because Docbook conditional statements are processed at the time the outputs are built, it causes issues with localization. This is because it is not obvious to translators what will be visible to readers, and where it will go. This is the same problem with entities, which are also only resolved when the outputs are built. Therefore, the use of Docbook conditional statements is discouraged.
PressGang CCMS Conditional Statements
PressGang CCMS conditional statements look similar to Docbook conditional statements, but with a few key differences.
  • Regular expression matching means that you can use fuzzy matching for conditionals. If something applies to every version of EAP, you can match on EAP.*. If it only applies to EAP 6, you can match on EAP6.
  • Conditions are resolved before content is sent for localization. This means that translators see exactly what needs to be translated, and there is no mystery for them to solve.
  • If you export content with conditional build statements to another system, the conditional statements are already resolved. This improves compatibility.

4.6.2.3. Build a Conditional Publication

To build a book using conditional statements, you need to edit the content specification. You can apply your conditional statements universally or on a per-topic basis. Note that the conditions are resolved before the payload is generated, so the XML that you ultimately build and publish does not contain any conditions. This means that you can send the strings for translation without worry.

Example 4.11. Apply Conditional Statements to the Whole Book

You can use an exact conditional or a pattern-matched one. The following example shows each type. This is applied in the top portion of the content specification. 
Title = Widget Installation Guide
Subtitle = The Most Efficient Way to Install Widgets
Abstract= A guide to installing widgets
Product = Widgets
Version = 2
Edition = 1
DTD = Docbook 4.5
Copyright Holder= Acme, Inc.

[condition=widgetsv1]
 
Chapter: Installation
  Section: Downloading
    ...
This will include all content which has no condition assigned to it, and any content which is tagged with the condition widgetsv1.
You can also include regular expressions including logical operators. So if I wanted to match either the condition "widgets*" (anything tagged with a conditional includes the string "widgets") or "newbie", it would be "widgets*|newbie". I could also ensure that both are matched, by using "widgets*&newbie".

Example 4.12. Apply Conditions to a Single Topic

You can apply a condition to a single topic, rather than the entire content specification. Regular expressions still work here. 
...
Chapter: Installation
  Section: Downloading
    Downloading the product [123, condition=newbie]
    Extracting the product [124]
    ...
Topic 123 will only include unconditional content or content which is tagged with the newbie condition.

4.6.2.4. Conditional Build Best Practices

Following are some guidelines for using conditional build statements without causing maintenance issues.
If you need to change someone else's topic, do use conditionals.
If you need to make significant changes to a topic that is used in another project, tag your changes with conditional build statements. This way, the other build is not impacted at all by your edits. If you feel that the topic itself needs to change for non-project-specific reasons, talk to the person who has made the most recent edits, or the team lead for the other project using the topic. They may want to use your changes too.
Projects which work together should agree on standards for conditional tags.
At the business unit level (Middleware, Cloud, Platform, etc) it is best for teams to agree upon a standard conditional build vocabulary. This should be located in a common place such as Docspace, for reference by everyone. This helps ensure that content is reusable and maintainable. It also means that regular expression matching is more likely to work as expected.
Tag the smallest thing possible.
If you only need to change one or two words in a paragraph, use a <phrase> and only tag those words. If you only need to change one paragraph in a section, tag only that paragraph. This helps maintenance and source XML readability.

Generally, localization does not happen in the testing phase, because feedback and corrections of documentation are still ongoing.
String Freeze
At the end of the testing phase, the documentation team will initiate string freeze. This means that no further changes are allowed to be made to the English version of the documentation. At this time, the documentation team lead freezes the content specification and pushes all strings to Zanata for translation, using the CSProcessor command. Zanata performs a provisional translation automatically, using its translation memory facility.
Publication
The English documentation is released in the product publication phase. Any translations that are required at product GA are also published. Generally, localization work continues for weeks or months after product GA.
Translation Tools
The translation team works from DocBuilder. In the localized instances of DocBuilder, the equivalent of the Edit This Topic link opens in Zanata, rather than the PressGang CCMS Web UI. In this way, translators can see their work in context, and do not need to build books locally.

6.1. Suggested Migration Workflow

Migrating content to PressGang CCMS is not yet automated. However, enough content has been migrated that we have some information on patterns and workflows that seem to work. These guidelines apply whether your current content is in Docbook, DITA, LibreOffice, HTML, or anything else.

Procedure 6.1. General Migration Workflow

  1. Normalize your content.

    Go through your content and make it as topic-based as possible. This may involve some rewriting. Generally, it is a good idea to do this the release before you plan to actually migrate the content. Be sure that your content is broken up into a hierarchical structure of parts, chapters, sections, appendixes. Try to get rid of "glue" content which is unnecessary.

  2. Create a content specification map representing your content.

    This step may be done before or after the next step. People have different preferences. Creating a content specification based on your existing content will allow you to automatically create blank topics to hold the content you are importing. This will also allow you to see the structure of your content, and check to be sure that it is structured appropriately.
    Currently, the content specification is a human-and-machine-readable text file with special syntax. Eventually, this will be a GUI interface which allows you to drag and drop to reorganize content.

  3. Extract each topic from your existing content.

    You can do this before or after you create the content specification. Save each topic as a new topic in the CCMS. If you do it before creating the content specification, you will need a way to track the topic IDs themselves as you go. It is often easier to start with the content specification.
    If you are importing from a non-Docbook format, you will need to convert the content to Docbook. Any document that LibreOffice can open can be exported to Docbook. You can use the application pandoc to export HTML or MarkDown to DocBook. You can use the application asciidoctor to convert AsciiDoc to Docbook. As this topic is only an overview, details for these conversions will be provided later.

  4. Import images.

    Images in your document need to be imported into PressGang CCMS. Generally, this is done as a bulk import. THe original file names of the images will be saved during the import, and you can search for them to find the image IDs for the next step.

  5. Clean up errors and mistakes in the imported book.

    Inevitably, there will be errors in your content. For instance, <xref> elements need to be converted into injections. Image links need to be modified to use the image IDs that are imported into PressGang CCMS.

  6. Import old translations into Zanata.

    If old translations for your content were not already in Zanata, you need to import them. To do this, you will need to contact the Zanata team to create a read-only project with your old content's translations. This will feed the strings into the translation memory, and will allow your imported content to be at least partially translated, so that old translations are not lost.

6.2. Get Content Ready to Move

Getting your content ready to move may entail multiple passes, in order to make the migration as smooth as possible. When you are finished with this process, your content will meet the following criteria:

Checklist for Migration Readiness

  • Content is in Docbook (currently Docbook 4.5 is strongly recommended)
  • Content is organized into a hierarchical structure of parts, chapters, sections, appendixes.
  • At the lowest level, sections are identified roughly as concepts, tasks, and references.
  • Elements such as <sect1> and <simplesect> are not used.
The checklist items need not be implemented in the order they are given. It may be easier for you to reorganize your content before converting it to Docbook, for instance. Following are some tips and guidelines.
Content in LibreOffice or Microsoft Office Formats
If your content is currently in a word processor format, the best thing that you can do for yourself is to create and use a consistent set of styles for your content. Organize your content hierarchically using header levels, and use styles rather than applying fonts, text sizes, and colors manually. This may take quite a bit of up-front work, but it will save you work in the long run.
LibreOffice includes an ability to export your content to DocBook. This goes much more smoothly if you have been consistent with formatting styles.
You should also reorganize your content to adhere to a topic-based approach, as much as possible. Attempt to identify concepts, tasks, and references. Tasks and references may be buried inside paragraphs. When it is appropriate, turn them into procedures, tables, or lists.
Content in Confluence Wiki Format
The JBoss.org instance of Confluence Wiki has the ability to export content as Docbook, but it does not work well. However, you can copy Confluence output and paste it directly into LibreOffice. At that point, you can follow the steps outlined for dealing with LibreOffice content.
Content in HTML
If your content is in HTML, you can use a command-line utility called pandoc to convert it to DocBook. pandoc is available for Red Hat Enterprise Linux, Fedora, Mac OSX, and Windows systems.
Content in PDF
If your content is in PDF, you can use the PDF Import feature of LibreOffice to convert your PDF to LibreOffice format, and follow the LibreOffice import guidelines from there. This will not work well if the PDF was created from images. You can tell if your PDF was created from images if you cannot select text in the PDF. In this case, your best hope is in an OCR application, to try to convert text in images into plain text. This would be very difficult to convert to DocBook.
Content in Docbook 5
PressGang CCMS is DTD-agnostic, meaning that it does not actually notice or care which XML format your content is in. However, in order to build outputs, your content needs to be compatible with Publican. Publican 3 has some support for Docbook 5, so you may be able to use your content as-is. However, if you run into trouble, you may need to down-covert your content to Docbook 4.5 first.
Content Already in DocBook 4.5
If your content is already in Docbook, your conversion will be relatively easy. Go through your content and be sure that it is organized roughly into topics. It can be useful to use XML comments to mark topic boundaries and which type each topic maps to.
Tasks and references are often buried in paragraphs, when they would be more effective as procedures, lists, or tables. It is best to do this conversion as you go. You may find that the concepts you find are very short and not fleshed out. You may find gaps in your content. You can use XML comments or DocBook <remark> tags to make notes for yourself or your team.
Finally, search your content for problematic tags, such as <sect1>, <sect2>, and the rest of those tags, all the way to level 6. Another tag you should not use is <simplesect>. Substitute each of these tags with <section>.
Dealing with Images
Each of the conversion tools and processes above should save your images to one or more directories. If your images are not in JPG, GIF, PNG, or SVG format, you should convert them using the image manipulation tool of your choice. SVG or PNG are the preferred formats for image quality and compatibility.

Warning

Maven and jDocBook support are provided as technology preview features. If you run into trouble or you require different behavior, submit bugs or RFEs as appropriate.
PressGang CCMS provides technology-preview level support for adding Maven and jDocbook artifacts to your book, so that you can work with an upstream project that requires Maven and jDocBook support for documentation. Such projects typically need the ability to do nightly or continuous-integration builds in an external system such as Jenkins or Hudson.

Procedure 7.1. Add Maven and jDocBook Support

  1. Add the required metadata to your content specification.

    Add the Group ID and Artifact ID information to the top section of your content specification. Get this information from the upstream project team. Maven variables such as ${translation} are supported.

    Example 7.1. Example Group and Artifact ID settings

    GroupId = org.jboss.tools
ArtifactId = Red_Hat_JBoss_Developer_Studio_Getting_Started_Guide-en-US

  2. Be sure that your book's information is accurate and appropriate so that automatic settings can be determined.

    Several settings are determined automatically from your book's other metadata.
    • The <version> in the POM is populated using the Version with -SNAPSHOT amended.
    • The <docname> and <bookname> properties are set using the book Title, and <docname> uses the escaped Title. These are not currently customizable, because they are only used in the pom.xml, to set the <artifactId> and <name> elements, and for build or output parameters.
    • Two addition property fields have been added, so that they can be used if needed. These are <docproduct> and <bookproduct>, which are set in the same way as <docname> and <bookname>, but use the Product field.
    • The <corpauthor> element is removed from the Book_Info.xml to avoid issues with the build formatting.
    • The Common Content is copied from the Publican installation directory and placed inside the book automatically, whereas normally this would be done using override statements. If your Publican brand includes these files, they are used in preference to the ones present in the common (default) brand.

  3. Request the book output payload from the CSProcessor.

    To request a build or a preview of your book from the CSProcessor, use one of the following commands: 
    csprocessor build BOOK_ID --format jDocbook
    csprocessor preview BOOK_ID --format jDocbook

  4. If desired, override the POM file.

    You can override the default pom.xml when you create the payload with the CSProcessor, by appending the --override pom.xml=/path/to/pom.xml parameter.
